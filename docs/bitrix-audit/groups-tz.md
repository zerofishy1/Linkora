Дата live-аудита: 13 апреля 2026

Секреты в проект не сохранялись. Мутационные действия вроде вступления в группу или финального создания проекта не подтверждались, чтобы не менять рабочие данные.

## Главное поведение раздела

1. Список групп не является тупой таблицей.
2. `Создать` и клик по названию группы открывают `side slider` поверх текущей страницы.
3. Фоновая страница со списком остается под слайдером видимой и не размонтируется.
4. При этом URL в адресной строке меняется на целевой маршрут, а контент реально живет внутри iframe в слайдере.
5. Переход по названию группы ведет не всегда в одну и ту же вкладку.
   Для разных групп открывается их текущий landing:
   `Автогарант` открылся как лента,
   `Гермес1` открылся как задачи,
   `hh` открылся как диск.

Это важно повторить в клоне: у группы должен быть не один фиксированный роут, а настраиваемая стартовая секция или last-visited landing.

## 1. Страница списка групп

Маршрут:

- `https://c-community.bitrix24.ru/workgroups/`

Заголовок:

- `Группы и проекты`

### Визуальная структура

- верхняя панель раздела с кнопкой `Создать`
- строка быстрых фильтров и поиска
- счетчики над таблицей
- таблица групп
- пагинация и итоги внизу

### Что есть на экране

- кнопка `Создать`
- активный пресет `Активные`
- поисковая строка
- счетчики:

  `Коммуникации`,

  еще один блок `Коммуникации`,

- таблица
- чекбоксы выбора строк
- нижняя статистика:
  `ОТМЕЧЕНО: 0 / 15`,
  `ВСЕГО: 15`,
  `СТРАНИЦЫ: 1`,
  `ПРЕДЫДУЩАЯ`,
  `СЛЕДУЮЩАЯ`

### Колонки таблицы

- `Дата последней активности`
- `Участники`

### Содержимое строк

Каждая строка содержит:

- внизу есть пагинация

## 2. Кнопки списка и куда ведут

### Прямые переходы из таблицы

- название группы `hh` -> `/workgroups/group/3/`
- название группы `Автогарант` -> `/workgroups/group/23/`
- название группы `Гермес1` -> `/workgroups/group/33/`

### Кнопки действий

- `Создать` -> `/company/personal/user/19/groups/create/`
  Открывается через side slider.
- `Вступить` для закрытой группы `hh` -> `/workgroups/group/3/user_request/`
  Это мутационный сценарий, в audit не подтверждался.
- `Вступить` для открытых групп визуально доступен прямо из списка.
  Внутри открытого проекта также есть CTA `Присоединиться к проекту`.

### Важный UX-нюанс

Bitrix не уводит пользователя со списка в "новую страницу".  
Открытие группы и создание группы происходит поверх списка, как рабочий overlay-режим.

## 3. Side slider "Создать"

Маршрут после клика:

- `/company/personal/user/19/groups/create/`

Iframe внутри slider:

- `/company/personal/user/19/groups/create/?IFRAME=Y&IFRAME_TYPE=SIDE_SLIDER`

### Что видит пользователь

Wizard из шагов:

- `Тип`
- `Возможности`
- `Конфиденциальность`
- `Участники`

Первый экран показывает выбор типа:

- `Проект`
- `Коллаба`
- `Группа`

### Кнопки wizard

- `Продолжить`
- `Назад`
- `Сменить тему`
- скрытый/альтернативный action `Создать коллабу`

### Выявленные поля и настройки

- `GROUP_NAME`
- `GROUP_DESCRIPTION`
- `PROJECT_DATE_START`
- `PROJECT_DATE_FINISH`
- теги
- выбор исполнителя по умолчанию
- переключатели инструментов:
  `Задачи`,
  `Календарь`,
  `Диск`,
  `Чат`,
  `Сообщения`,
  `Фотогалерея`,
  `Маркетплейс`,
  `База знаний`
- признак `Для публикации`
- блок `Уровень конфиденциальности`
  с вариантами:
  `Открытый`,
  `Закрытый`,
  `Секретный`
- блок участников:
  владелец,
  руководитель / владелец продукта,
  скрам-мастер,
  модераторы,
  помощники руководителя,
  команда,
  заинтересованные лица

### Логика wizard

- шаг 1: выбрать тип проекта
- шаг 2: заполнить базовую карточку и набор возможностей
- шаг 3: выбрать конфиденциальность
- шаг 4: настроить участников
- финальный CTA должен создавать сущность

Для клона это значит:

- нужна многошаговая форма, а не один длинный экран
- нужно сохранить выбор типа и показывать разные подписи:
  `О проекте`, `О группе`, `О Скрам-команде`
- поля и набор участников должны меняться от типа

### Что видно

- кнопка `Вступить`
- кнопка `О проекте`
- вкладки:
  `Диск`,
  `Лента`,
  `Еще`

### Особенности экрана

- `Добавить` на диске disabled
- доступны `Подключить к диску` и `Корзина`
- кнопка вступления указывает на `/workgroups/group/3/user_request/`
- проект закрытый, поэтому вступление должно идти через запрос

### Вывод по логике

- closed-state нельзя делать таким же, как open-state
- часть инструментов может быть видима, но ограничена по правам
- состояние прав должно влиять и на CTA, и на disabled controls

## 7. Что нужно реализовать в нашем клоне

### Обязательная архитектура

- страница-список групп как базовый фон
- side slider поверх списка
- вложенный routed-content внутри slider
- сохранение списка под overlay
- group landing state

### Сущности

- `group`
- `group_access_state`
  значения:
  `member`,
  `open_non_member`,
  `closed_non_member`,
  `secret_non_member`
- `group_tools`
- `group_members`
- `group_owner`
- `group_tabs_state`
- `group_last_open_section`
- `group_privacy`

### Список групп

Нужно поддержать:

- таблицу с чекбоксами
- колонки из живого Bitrix
- бейдж действия справа
- аватарки участников
- фильтры и пресеты
- поиск
- пагинацию
- нижнюю статистику

### Create flow

Нужно поддержать:

- 4-шаговый wizard
- выбор типа
- описание и тему
- включение/отключение инструментов
- конфиденциальность
- подбор участников по ролям
- финальное создание без layout-сдвигов

### Group workspace

Нужно поддержать 3 разных сценария:

- открытый проект для неучастника
- закрытый проект для неучастника
- проект участника

Именно это определяет:

- доступные вкладки
- первичную CTA
- disabled-состояния
- доступ к контенту и инструментам

## 9. Что не кликалось намеренно

Чтобы не изменять рабочий портал, в audit не подтверждались:

- финальное вступление в проект
- отправка запроса на вступление
- финальное создание группы/проекта
- действия из `Действия`, если они меняют данные

Для реализации клона это не мешает, потому что:

- целевые URL и состояние кнопок собраны
- тексты CTA собраны
- доступные режимы и права собраны
- визуальная и навигационная схема подтверждена live

## 10. Ключевой вывод

Раздел `Группы` в Bitrix24 нельзя делать как "еще одну обычную страницу списка".  
Это связка из:

- каталога групп
- overlay-навигации
- мастера создания
- мини-workspace каждой группы
- разной логики доступа для участника и неучастника
- разных landing-ов для разных групп

Если переносить раздел качественно, то нужно повторять не только визуальные блоки, но и этот navigation model.
